# SSPS Project Coding & Review Rules v1.0

본 규칙은 SSPS 프로젝트의 구조적 안정성과 데이터 무결성을 보장하기 위해 모든 개발 및 수정 단계에서 반드시 준수해야 하는 강제 규정입니다.

## 1. 아키텍처 우선 원칙 (Architecture First)
- 모든 수정은 모듈의 역할(Role)과 의존성(Dependency)을 먼저 파악한 후 수행한다.
- 단순히 에러를 없애기 위한 임시방편 수정(Patchwork)은 금지한다.
- 변경 사항이 전체 시스템 구조에 미치는 영향을 검토하고, 필요한 경우 설계 문서를 먼저 업데이트한다.

## 2. 코드 구조 및 가독성 (Code Structure)
- **명시적 임포트**: 필요한 모든 모듈과 클래스는 파일 최상단에 명시적으로 임포트한다. (함수 내 지연 임포트는 순환 참조 해결 등 특수한 경우 외 금지)
- **철저한 예외 처리**: 외부 API(네이버, 올리브영 등) 및 DB(Supabase) 호출 시 반드시 `try-except` 블록과 로깅을 포함한다.
- **주석 준수**: 기능 변경 시 관련 주석을 즉시 업데이트하며, 특히 비즈니스 로직의 이유(Why)를 명시한다.

## 3. 데이터 무결성 규칙 (Data Integrity)
- **No Mock Data**: 어떠한 상황에서도 가짜 데이터를 생성하거나 하드코딩된 폴백 값을 사용하지 않는다. 데이터 부재 시에는 `pending` 혹은 `no_data` 상태를 정직하게 반환한다.
- **원본 보존**: API 원본 데이터는 가공 단계 이전에 반드시 별도 테이블(`trend_raw`)에 저장한다.

## 4. 리뷰 및 검증 절차 (Review Process)
- **셀프 리뷰**: 코드 수정 제안 전, AI 스스로 위 규칙을 준수했는지 다시 한번 검토한다.
- **리뷰어 도구 활용**: 복잡한 로직 변경이나 구조적 수정 시에는 `TestSprite` 등 분석 도구를 사용하여 무결성을 검증한다.
- **사용자 승인**: 모든 코드 적용은 구조적 타당성에 대한 충분한 설명과 함께 사용자에게 제안하고 승인을 받는다.

## 5. 수익성 및 UX 최적화 (Monetization & UX)
- 모든 상품 링크 및 검색 연결은 `Coupang Partners` 수익화를 우선하도록 설계한다.
- UI는 사용자에게 불필요한 노이즈(레거시 문구, 동의창 등)를 최소화하고 데이터에 집중할 수 있도록 단순함을 유지한다.
