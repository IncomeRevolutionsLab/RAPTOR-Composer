# SSPS (Intelligent Shopping Short-form Product Selection System) PRD v3.6

## 0. 프로젝트의 기원 (Project Genesis)
*이 섹션은 프로젝트 초기 사용자님이 제시한 원형의 가치를 보존하기 위해 작성되었습니다.*

- **최초 구상**: "쇼핑 숏폼의 소재가 되는 상품을 선정하는 것은 매우 고된 작업이다. 이를 데이터 기반으로 자동화할 수 없을까?"
- **초기 선정 기준**:
    - **지금 핫한 상품**: 드라마, 뉴스, SNS 등 외부 요인으로 급상승하는 키워드.
    - **계절적 반복 수요**: 여름 수영복, 명절 선물 등 매년 같은 시기에 터지는 상품.
    - **스테디셀러**: 일년 내내 생활 속에서 꾸준히 팔리는 필수 아이템.
- **최초 목표**: 네이버 데이터랩의 모든 카테고리를 뒤져서, 인간의 직관을 보완하고 데이터에 기반한 '진짜 팔릴만한 상품'을 골라내는 엔진을 만드는 것.

## 1. 프로젝트 목적 및 배경 (Purpose & Background)
본 프로젝트는 다중 쇼핑 플랫폼의 데이터를 통합 분석하여 숏폼 콘텐츠 제작에 최적화된 상품을 선정하는 지능형 의사결정 시스템입니다.

- **핵심 목표**: 네이버 데이터랩의 5,807개 세세분류 데이터를 자동 분석하여 인간의 한계를 넘는 효율적인 상품 선정 및 수익화(Coupang Partners) 연결.
- **철학**: 데이터의 선후 관계와 상대적 비중을 보존하며, 절대 가짜 데이터를 사용하지 않는 **정직한 분석(Honest Analytics)**.
- **연결**: 랩터 쇼핑숏폼개발기와의 연동을 통해 [분석-선정-기획-제작]의 전 과정을 자동화함.

## 2. 분석 대상 상품의 정의 (Target Products)
1. **초급상승형 (Hot Items)**: 최근 검색량 급상승 (드라마 유행, 특정 기사, SNS 이슈 등).
2. **시즌 반복형 (Seasonal Items)**: 매년 반복되는 계절적 요인 및 기념일 수요 (여름 바캉스 용품, 명절 선물, 크리스마스 등).
3. **스테디셀러 (Steady Sellers)**: 생활 필수품 등 일년 내내 꾸준한 수요가 발생하는 상품군.

## 3. 데이터 아키텍처 및 수집 전략 (Data & Collection)

### 3.1 네이버 쇼핑인사이트 API 수집 규격
- **수집 단위 (Batching)**: 1회 요청 시 최대 **3개**의 주제어(Category ID)를 묶어서 수집. (API 제한 사항 준수)
- **비교 기준 (Anchoring)**: 배치 간 상대적 크기 비교를 위해 '그룹 내 앵커 상품'을 포함하여 정규화 수행.
- **수집 주기**: 일일 1,000회 API 한도를 활용하여 약 5,800개 카테고리를 이틀(48시간) 주기로 전체 갱신.

### 3.2 롤링 업데이트 (Rolling Update) 정책
- **기존 방식 (Atomic Swap)**: 98% 수집 완료 시까지 구버전 유지 후 한꺼번에 교체. (폐기)
- **신규 방식 (Rolling Update)**: 수집되는 즉시 활성 버전(Active Version)에 실시간 반영. 
  - 50%는 최신 데이터, 50%는 전일 데이터인 상태라도 서비스 중단 없이 최신 정보를 제공함.
  - 사용자에게는 "실시간 수집 및 반영 중"임을 투명하게 공지.

### 3.3 원본 데이터 보존 (Raw Data Archive)
- 가공된 트렌드 지수 외에도, 네이버 API가 반환한 **가공 전 원본 JSON**을 `trend_raw` 테이블에 무조건 보존함.
- 목적: 분석 로직 오류 검증 및 향후 디버깅을 위한 감사 데이터(Audit Trail).

## 4. 수익화 및 사용자 경험 (Monetization & UX)

### 4.1 쿠팡 우선 검색 연결 (Coupang-First)
- **수익 모델**: 분석된 상품을 즉시 쇼핑으로 연결하여 쿠팡 파트너스 수익 창출.
- **기본 설정**: 분석 결과 및 키워드 클릭 시 **쿠팡 검색 결과**로 우선 연결함.
- **사용자 선택**: 필요 시 네이버 쇼핑 검색으로 전환할 수 있는 옵션 제공.

### 4.2 UI/UX 원칙
- **불필요 요소 제거**: 법적 약관 동의 문구, 레거시 수동 입력 필드 등 분석에 방해되는 요소를 배제하여 깔끔한 대시보드 유지.
- **투명한 상태 보고**: 데이터가 없는 경우 가짜 값을 보여주지 않고 "수집 중" 혹은 "분석 준비 중"임을 명확히 표시.

## 5. 시스템 리포팅 (Reporting)
- **텔레그램 리포트**: 매일 1회, 수집 엔진의 상태를 사용자에게 보고.
  - 오늘 신규 수집 건수
  - 전체 카테고리 대비 진행률 (%)
  - 서버 생존 확인 (Supabase DB 활성화 체크)

## 6. 기술 스택 (Tech Stack)
- **Backend**: Python, Flask, Render (Serverless/Web Service)
- **Database**: Supabase (PostgreSQL)
- **Frontend**: Vanilla JS, ECharts-GL (3D 시각화), Lucide Icons
- **AI**: Gemini 1.5 Pro (RAPTOR GEM 연동용)
