# SSPS Project History & Evolution

이 문서는 SSPS 프로젝트의 초기 구상부터 현재 v3.6에 이르기까지의 개발 여정과 주요 의사결정 맥락을 기록합니다.

## 1. 초기 구상 및 비전 (Initial Vision)
- **최초 프롬프트 의도**: "수많은 쇼핑 상품 중 어떤 것이 숏폼의 소재로 적합할까?"라는 질문에서 시작.
- **핵심 요구사항**:
    - 매일 검색되는 방대한 네이버 쇼핑 트렌드를 사람 대신 분석할 것.
    - 유행 상품, 시즌 상품, 스테디셀러를 체계적으로 분류할 것.
    - 분석된 결과를 숏폼 제작 파이프라인(RAPTOR)으로 즉시 연결할 것.

## 2. 버전별 진화 과정 (Version Evolution)

### Phase 1: 기반 구축 (v1.0 ~ v2.0)
- **주요 내용**: 네이버 데이터랩 API 연동 및 기본 4단계 카테고리 트리 구축.
- **발전 사항**: 단순 검색어 조회를 넘어 DB(Supabase)를 활용한 트렌드 캐싱 체계 마련.

### Phase 2: 구조 고도화 및 시각화 (v2.1 ~ v3.0)
- **주요 내용**: 3D ECharts-GL을 활용한 입체적 트렌드 시각화 도입.
- **발전 사항**: 
    - Closure Table 스키마를 통한 계층적 카테고리 탐색 최적화.
    - 앵커링(Anchoring) 로직 도입으로 서로 다른 배치 데이터 간 비교 가능성 확보.

### Phase 3: 정직한 데이터 및 수익화 (v3.1 ~ v3.6)
- **주요 내용**: 'No Mock Data' 정책 수립 및 롤링 업데이트(Rolling Update) 도입.
- **향상된 점**:
    - **정직함**: 가짜 데이터를 완전히 배제하고 `pending` 상태를 투명하게 공개.
    - **수집 효율**: 네이버 API 3개 배치 수집을 통해 48시간 내 전 카테고리 갱신 달성.
    - **수익성**: 분석 결과를 쿠팡 파트너스 링크와 직접 연결하여 실질적 수익 모델 확보.
    - **안정성**: 텔레그램 봇을 통한 일일 수집 현황 자동 보고 체계 구축.

## 3. 핵심 의사결정 맥락 (Key Decision Context)

### 왜 Atomic Swap 대신 Rolling Update인가?
- 사용자는 98% 완성을 기다리는 것보다, 50%라도 최신 데이터를 즉시 확인하고 활용하는 것이 실용적이라고 판단함. (2026.05.01 결정)

### 왜 쿠팡 우선인가?
- 본 시스템의 궁극적 목적은 단순 분석이 아닌 '수익을 창출하는 숏폼 제작'에 있으며, 쿠팡 파트너스는 가장 즉각적인 수익화 수단이기 때문임.

### 왜 원본 데이터를 보존하는가?
- 가공된 지수는 분석 로직에 따라 변할 수 있지만, 네이버가 제공한 원본 값은 변하지 않는 '진실의 근거'가 되기 때문임 (디버깅 및 신뢰도 확보).
