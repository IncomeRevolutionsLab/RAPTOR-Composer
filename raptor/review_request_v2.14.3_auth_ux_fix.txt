현재 변경된 상태를 실제 소스 코드(src/components/AuthDashboard.tsx)와 교차 검증하여 아키텍처 사후 리뷰(Post-Review) 보고서를 한국어로 작성해줘.

[이번 패치 배경]
기획자 결정에 따라 이메일 인증 방식을 유지하기로 확정되었으나, 이에 따른 3가지 UX/법적 결함이 발견되었다:
1. (P1-법적) 회원가입 버튼 주변에 약관 동의 문구가 없어 법적 고지 누락
2. (P1-UX) 회원가입(signUp) 성공 시 화면에 아무 피드백이 없어 사용자가 가입 완료 여부를 알 수 없음 (무반응 버그)
3. (P1-UI) 화면 크기가 작을 때 에러/성공 메시지 박스가 잘려 보이는 UI 결함

[검토 대상 변경 사항 요약]

변경 1: src/components/AuthDashboard.tsx — 회원가입 성공 UX 수정 (P1-UX)

  변경 전 구조 (버그):
    // 회원가입 성공 시 즉시 setUser(data.user) + setIsModalOpen(true) 호출
    // → 이메일 미인증 상태로 대시보드 진입 허용 (이메일 인증 무력화)
    // → 성공 피드백이 authSuccess 상태에 "회원가입 성공!" 저장되나
    //    setIsModalOpen(true)가 즉시 실행되어 인증 안내 화면을 덮어쓰는 구조
    setUser(data.user);
    setAuthSuccess("회원가입 성공!");
    setPassword('');
    setTimeout(() => setIsModalOpen(true), 500);

  변경 후 구조 (수정):
    // setUser/setIsModalOpen 제거 → 인증 메일 발송 안내만 표시
    // 이메일 인증 완료 전까지 로그인 불가 → 이메일 인증 흐름 정상화
    setAuthSuccess("가입하신 이메일로 인증 메일이 발송되었습니다. 메일함의 인증 링크를 클릭하여 가입을 완료해 주세요.");
    setPassword('');
    setEmail('');

[리뷰 포인트]
1. 이메일 인증 흐름 완결성:
   - 회원가입 후 setUser()를 호출하지 않으므로 미인증 사용자가 대시보드에 진입하는 경로가 완전히 차단되었는가?
   - Supabase의 signUp 응답에서 이메일 미인증 시 data.user.email_confirmed_at이 null인데, 이를 체크하는 가드가 로그인 경로에도 적용되어 있는가? (백엔드 /api/auth/signin의 email_confirmed_at 검증 로직)
   - 회원가입 후 이메일 미확인 상태로 로그인 시도 시 사용자 경험은 어떻게 처리되는가?

2. 약관 동의 고지의 법적 충분성:
   - 현재 구현: "이용약관 및 개인정보 처리방침에 동의하는 것으로 간주됩니다." 텍스트만 표시 (링크 없음)
   - 실제 이용약관 및 개인정보처리방침 페이지가 없을 경우 법적 리스크는?
   - UI/UX 관점에서 클릭 가능한 링크가 없는 것의 개선 권고사항

3. 에러 메시지 UI 완결성:
   - items-start 변경으로 아이콘이 상단 정렬됨 → 짧은 에러 메시지 시 아이콘-텍스트 세로 정렬 불일치 가능성
   - style 인라인 객체(wordBreak, overflowWrap)와 Tailwind break-words 클래스 중복 적용의 적절성

4. 회원가입 성공 후 UX 흐름:
   - 성공 메시지 표시 후 이메일/비밀번호 폼이 클리어됨 → 사용자가 이미 가입된 이메일을 다시 입력해 중복 가입 시도 시 에러 처리 경로 확인
   - authSuccess 상태 표시 후 자동 해제 타이머가 없음 → 모달을 닫고 다시 열면 성공 메시지가 사라지는가? (authSuccess는 null 초기화되는 시점 확인)
