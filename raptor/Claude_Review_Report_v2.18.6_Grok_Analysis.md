## 1. 기획자 제공 스펙 문서 vs 실제 소스코드 비교 분석

기획자님이 문의하신 **"grok imagine text - image"** 모델에 대하여, 제공해주신 JSON 예제와 현재 소스코드를 면밀히 분석한 결과 다음과 같은 중대한 모순점과 현황을 발견했습니다.

### ⚠️ 발견된 모순점 (Contradiction)
* 기획자님의 지시사항 제목은 **`text-image` (이미지 생성)** 이지만,
* 올려주신 JSON 스펙 내용의 `model` 명칭은 **`grok-imagine/text-to-video` (텍스트 기반 비디오 생성)** 이며, 파라미터 역시 `duration`(길이), `mode`, `resolution`(해상도) 등 전형적인 **비디오 전용 파라미터**들로 구성되어 있습니다.

### 🟢 `main.py`의 현재 상태 (100% 정상 및 호환 완료)
만약 기획자님이 의도하신 바가 **이미지 생성(`text-to-image`)** 이 맞다면, 현재 `main.py`는 이미 완벽하게 조치되어 있습니다.

방금 전 GPT 이미지 모델 픽스를 진행하면서 공통 페이로드 생성부를 정규화했기 때문에, 현재 유저가 "grok" 엔진을 선택하여 이미지 생성을 요청하면 아래와 같이 순수한 스펙으로 전송됩니다.
```json
{
    "model": "grok-imagine/text-to-image",
    "input": {
        "prompt": "유저의 프롬프트",
        "aspect_ratio": "auto" (또는 지정된 비율)
    }
}
```
위 페이로드는 영상 전용 쓰레기 파라미터(`duration`, `quality`, `image_url` 등)가 완벽히 소거되어 있으므로 422 에러 없이 **정상 작동을 보장**합니다.

### 🔴 사용되지 않는 `text-to-video` 엔드포인트
반면 올려주신 스펙이 **텍스트에서 바로 영상을 뽑아내는 `text-to-video`** 모델이라면, 현재 RAPTOR 파이프라인(항상 이미지를 먼저 뽑고 `image-to-video`로 애니메이팅하는 구조)에서는 해당 모델을 호출하는 로직 자체가 `main.py`에 존재하지 않습니다.

## 2. 리뷰어의 권고사항 및 질문

기획자님, 올려주신 JSON 예제는 명백히 **비디오 생성 모델(`text-to-video`)**의 스펙입니다. 다음 두 가지 중 어느 쪽에 해당하시는지 확인을 부탁드립니다.

1. **"내가 JSON을 잘못 가져왔다. 기존 `text-to-image`가 잘 동작하면 된 거다."**
   👉 현재 코드는 이미 완벽하게 정규화되어 수정할 곳이 없습니다. `v2.18.6 GPT & Grok 이미지 핫픽스`를 그대로 실서버에 배포하시면 됩니다.
2. **"RAPTOR에 `text-to-video` 모델을 새롭게 도입해야 한다."**
   👉 현재 파이프라인 아키텍처를 대대적으로 뜯어고쳐야 하므로 별도의 신규 개발 플랜(Implementation Plan)을 세워야 합니다.
