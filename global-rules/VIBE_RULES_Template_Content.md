🧾VIBE_RULES_Template_Content (콘텐츠/서비스 중심)
Development Rules — 협업·품질·배포 규칙 / 재사용 모듈 운영

1. 역할과 책임(사람/AI)
사람: 주제 선정, 사실검증, 최종 승인
AI: 초안 생성, 구조화, 메타/SEO, 썸네일 프롬프트, 리포트 초안
금지: AI의 임의 인용/허위수치 생성

2. 버전 관리
메인 가이드(Main V1.2)에 PLM 버전 종속(PLM-XX v1.2.x)
변경 로그 필수: Added/Changed/Removed/Fixed
파일명 규칙 준수: [Project]_[DocType]_V[Version].md

3. 품질 보증(QA) 및 이중(사전/사후) 리뷰 의무화
체크리스트: 사실검증, 표/그래프 설명문, 출처/저작권 표기, SEO 점수, CTA 포함
중복/표절 검사(요청 프롬프트 포함)
금지 표현 목록 관리(업데이트 시 공지)

🚨 [필수 강제] 사전 리뷰(Pre-Review) 의무화:
실행 계획서(implementation_plan.md 등) 작성 직후, 실제 코딩에 착수하기 전에 반드시 터미널에서 `claude -p` 명령어를 통해 Claude Code의 사전 아키텍처 리뷰를 받아야 한다. 클로드가 지적한 결함과 블로커를 계획서에 100% 반영(Overwrite)한 후에만 코딩을 시작할 수 있다.

🚨 [필수 강제] 사후 리뷰(Post-Review) 의무화: 
모든 코드 수정 및 자체 검증이 끝난 후, 임의로 작업을 종료할 수 없다. 반드시 로컬 터미널에서 `claude -p` 명령어를 통해 Claude Code의 사후 아키텍처 리뷰를 받아야 하며, 클로드가 아키텍처/보안/성능 측면에서 'PASS'를 선언한 결과 보고서를 제출해야만 마일스톤 완료로 인정한다.

4. 모듈 풀 운영 규칙
모듈 ID 부여, 입력/출력 계약(Contract) 문서화
재사용 기준: 동일 입력 스키마 준수 시 교체 가능
폐기/승격 정책: 실험 → 베타 → 정식
의존성: 모듈 간 호환 버전 표기

5. 자동화·배포
릴리즈 라벨: Draft → Review → Approved → Published
배포 로그: URL, 시각, 채널, 담당
실패 복구: 롤백 절차(이전 버전 복원, 수동 게시 대안)

6. 보안·정책
API 키 보호, 개인 정보 제거
플랫폼 정책 위반 방지(유튜브/애드센스 등)

7. 보고 & 회고
주간 리포트: 생산량, KPI, 이슈, 개선안
월간 회고: 모듈 성과, 교체/개선 대상 선정

8. 사용자 시나리오 맥락 유지 및 임기응변 금지 (Context Preservation)
- 맥락 유지: '문맥 다이어트'를 핑계로 비즈니스 시나리오의 핵심 단계나 기획 의도를 누락하는 행위를 엄격히 금지한다.
- 임기응변 금지: 특정 에러를 해결하기 위해 전체 아키텍처나 사용자 경험(UX)을 훼손하는 '땜빵식' 코드 수정을 절대 금지한다.
- 구조적 접근: 모든 수정은 상위 설계(PRD/MD)의 일관성을 먼저 확인한 뒤 진행한다.
