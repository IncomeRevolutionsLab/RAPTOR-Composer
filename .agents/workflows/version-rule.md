# SSPS 버전 표기 강제 규칙

> ⚠️ 이 워크플로우는 AI가 임의로 버전 번호를 창작하는 것을 방지하기 위한 강제 규칙입니다.

## 핵심 원칙

**버전 번호는 절대로 AI가 임의로 결정하지 않는다.**
버전 번호는 오직 코드베이스(README.md)에 명시된 값만 사용한다.

## 보고서/기획서 작성 전 필수 체크리스트

1. **Step 1: 현재 버전 확인 (필수)**
   - `c:\Antigravity Work\README.md` 를 열어 `공식 버전:` 항목을 확인한다.
   - 또는 `backend/engine/json_packager.py` 의 `schema_version` 값을 확인한다.

2. **Step 2: 다음 버전 결정 규칙**
   - 버그 픽스 / UI 소폭 수정 → **패치 버전 +0.1** (예: v1.3 → v1.4)
   - 새 기능 추가 / 알고리즘 고도화 → **마이너 버전 +1** (예: v1.3 → v2.0)
   - 아키텍처 전면 재설계 → **메이저 버전 +1** (예: v1.3 → v2.0)
   - **단, 버전 번호 변경은 반드시 사용자에게 먼저 확인 후 진행한다.**

3. **Step 3: 보고서 제목 작성 규칙**
   - 형식: `SSPS [확인된 버전 또는 다음 버전]: [작업 내용]`
   - 예시: `SSPS v1.3: 현재 상태 보고` 또는 `SSPS v1.4 (제안): QA 피드백 반영 계획`
   - 사용자 승인 전 "제안" 임을 괄호로 반드시 명시한다.

4. **Step 4: README.md Changelog 업데이트**
   - 작업 완료 후 README.md 의 버전 및 변경 이력 표를 반드시 업데이트한다.
   - `json_packager.py` 의 `schema_version` 값도 동일하게 수정한다.

## 금지 사항

- ❌ 코드 확인 없이 버전 번호를 감으로 결정하는 것
- ❌ "작업이 방대해 보이니까 major 버전을 올리겠다"는 자의적 판단
- ❌ 보고서 제목과 실제 코드 버전이 다른 상태로 제출

## 버전 불일치 발생 시 대처

만약 보고서 제목과 코드베이스 버전이 다름이 발견될 경우:
1. 즉시 사용자에게 불일치 내용과 원인을 보고한다.
2. 코드베이스 버전(README.md)을 기준으로 정정한다.
3. 잘못 작성된 보고서 제목을 수정하고 사용자에게 통보한다.
